# 班次小助手产品想法记录

## 产品定位

面向小型门店团队的微信小程序工具，重点解决人员入职、班次安排、门店打卡和日常管理。对外命名尽量使用“小助手”“工具”等中性表达，避免在小程序审核材料中过度使用敏感或容易被误解的行业词。

## 用户角色

- 管理员：维护门店、员工、班次规则，生成和调整班表，查看打卡记录。
- 员工：完成入职绑定，查看个人班次，按所在门店位置打卡。
- 测试场景：同一台手机可能既要模拟管理员又要模拟员工，需要保留测试角色切换入口；正式上线前可关闭。

## 员工入职与身份识别

- 管理员在员工维护页办理入职，生成入职码。
- 员工扫码或手动输入入职码完成注册绑定。
- 员工绑定时获取微信头像、昵称和手机号。
- 手机号正式上线需要后端或云函数用授权 code 换取，体验版可手动填写兜底。
- 员工完成绑定后，后续打开小程序自动进入员工端。
- 员工离职后，不再参与新班表生成，也不能继续打卡。

## 管理端能力

- 维护多个门店：名称、地址、经纬度、允许打卡半径。
- 维护员工：入职、离职、恢复、归属门店、岗位、手机号、每周最多班次数。
- 维护班次规则：早班、中班、夜班等班次所需人数。
- 自动生成本月班表。
- 手动调整某天某班次人员。
- 查看全店班表和打卡记录。

## 员工端能力

- 查看自己的班次，不展示无关人员安排。
- 选择或自动识别当前员工身份。
- 上班打卡、下班打卡。
- 展示当天打卡记录，包括门店、距离和时间。

## 多门店与定位打卡

- 系统支持多个门店。
- 员工新增时默认归属当前门店。
- 员工打卡时调用定位能力。
- 根据员工所属门店和当天排班门店，自动匹配最近的允许打卡门店。
- 只有距离在门店设置半径内才允许打卡。
- 打卡记录保存门店、经纬度、距离、时间。
- 正式审核前需要申请 `wx.getLocation` 权限，并在隐私协议中说明定位只用于门店打卡匹配。

## 后台服务想法

当前 MVP 以本地存储为主。正式使用需要后台或云开发承载数据：

- 用户表：openid、昵称、头像、手机号、角色、绑定员工。
- 员工表：姓名、手机号、岗位、状态、门店、入职码。
- 门店表：名称、地址、经纬度、打卡半径。
- 班次表：班次名称、时间段、所需人数。
- 排班表：日期、门店、班次、员工列表。
- 打卡表：员工、门店、班次、打卡类型、时间、位置、距离。

## 审核与命名注意

- 小程序名称和页面文案优先使用“班次小助手”“门店小工具”“团队助手”等表达。
- 避免在名称、简介和审核材料里突出“网吧”“排班系统”等可能影响审核的关键词。
- 定位权限申请理由要聚焦“员工打卡时匹配门店位置”。
- 提交审核前确保不是纯 demo，至少能正常打开、切换角色、查看班次、完成体验版打卡流程。

## 后续迭代方向

- 云开发或轻量后端接入。
- 手机号授权后端换取能力。
- 管理员邀请二维码。
- Excel 导入导出班表。
- 员工请假、不可排班日期。
- 夜班后自动休息约束。
- 按岗位匹配班次。
- 班次冲突检测。
- 门店维度统计报表。
- 打卡异常提醒和补卡审批。
